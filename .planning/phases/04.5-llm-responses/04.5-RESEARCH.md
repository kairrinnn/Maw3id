## Summary

Phase 4.5 inserts a Gemini-powered reply layer between the FSM transition engine and the `sendTextMessage` call. The FSM today returns `{ nextState, replyText }` where `replyText` is a hardcoded French string. The goal is to replace only `replyText` generation — the FSM logic, state transitions, and all booking/cancel/modify DB operations remain unchanged.

The architectural approach is a new async function `generateReply(context: ReplyContext): Promise<string>` that sits in `src/lib/llm/reply.ts`. The webhook route (`route.ts`) calls it after `processIntent()` resolves `nextState`, passing the transition context (current step, next step, service name, date, time, client name, salon name). If Gemini fails or times out, the function returns a pre-written French fallback string that mirrors the current hardcoded text. The FSM test suite must remain green: since `processIntent()` still returns `replyText` from hardcoded strings, the existing tests do not change. New tests cover the reply layer independently.

The only behavioral risk is latency. Gemini 2.5 Flash benchmarks at ~0.5s time-to-first-token and ~218 tokens/second output. A 60-token reply (typical WhatsApp message) takes roughly 0.5 + 0.3 = 0.8s under normal conditions. The 4s budget is comfortable, with room for 1-2s network overhead. Non-streaming (single `generateContent` call collecting the full reply) is the right choice here — WhatsApp does not stream to end users, so streaming adds complexity without benefit.

**Primary recommendation:** Add `src/lib/llm/reply.ts` with `generateReply(context)` that calls `ai.models.generateContent()` for plain text (no JSON schema). Wrap every call in a 3-second `Promise.race` timeout. On any failure, return a hardcoded fallback. Wire into `route.ts` after `processIntent()`, replacing the `replyText` variable only.

## Architecture Patterns

### Current State: FSM Returns Both Transition AND Reply
```
extractIntent() → processIntent() → { nextState, replyText (hardcoded) } → sendTextMessage(replyText)
```

### Target State: FSM Returns Transition, Gemini Generates Reply
```
extractIntent() → processIntent() → { nextState, replyText (hardcoded fallback) }
                                              ↓
                                  generateReply(context) → Gemini → replyText (warm, human)
                                              ↓ (on failure)
                                  replyText = hardcoded fallback from processIntent()
                                              ↓
                                  sendTextMessage(replyText)
```

### Recommended Project Structure Addition
```
src/lib/llm/
├── intent.ts          # unchanged — extractIntent()
├── models.ts          # unchanged — tryGemini() for JSON structured output
├── prompts.ts         # unchanged — intent system prompt
├── types.ts           # add ReplyContext interface here
└── reply.ts           # NEW — generateReply(), buildReplyPrompt(), FALLBACK_REPLIES map
```

### Pattern 1: ReplyContext — What to Pass Gemini

The reply prompt needs enough context to produce a contextually appropriate response without exposing sensitive data or requiring conversation history (which we deliberately do not store).

```typescript
// Source: codebase analysis of machine.ts + types.ts
export interface ReplyContext {
  currentStep: string        // where the FSM was before this message
  nextStep: string           // where the FSM is going (drives reply style)
  intent: string             // 'book' | 'cancel' | 'modify' | 'query' | 'greeting' | 'unknown'
  serviceName?: string | null
  date?: string | null       // ISO date e.g. "2026-04-09"
  time?: string | null       // "15:00"
  clientName?: string | null
  salonName?: string         // from bot_configs or hardcoded default
  conflict?: boolean         // true if slot is taken (conflict reply)
}
```

**What NOT to pass:** raw message history, internal booking IDs, tenant_id, pricing. Gemini only needs what a human customer service agent would need to write the reply.

### Pattern 2: System Prompt for Reply Generation

The reply system prompt is distinct from the intent extraction prompt. It is instructional (persona) not analytical (extract fields).

```typescript
// Recommended system prompt structure (not a call to build, just the pattern)
`Tu es l'assistante d'un salon de beauté marocain. Tu réponds en français avec une touche chaleureuse
marocaine (tu peux ajouter "inshallah", "mabrook", "wakha" de façon naturelle et rare — pas à chaque
message). Tes réponses sont courtes (1-3 phrases maximum), chaleureuses, et directes.
Tu ne dois jamais mentionner que tu es une IA.
Salon: ${salonName}
`
```

### Pattern 3: Per-Transition Reply Hints

Each FSM transition has a distinct communicative goal. The user prompt (not system) tells Gemini what kind of reply is needed:

| nextStep | Reply Goal | Example hint in user prompt |
|----------|-----------|----------------------------|
| `awaiting_service` | Ask which service | "Le client veut réserver. Demande-lui quel service il souhaite. Services: [liste]" |
| `awaiting_datetime` | Service selected, ask date+time | "Le client a choisi [service]. Demande-lui quand il souhaite son rendez-vous." |
| `confirming` | Confirm the booking details | "Confirme les détails: [service] le [date] à [time]. Demande oui ou non." |
| `confirmed` | Booking is done | "La réservation est confirmée: [service] le [date] à [time]. Félicite le client." |
| `cancelling` | Cancellation acknowledged | "Le client annule son rendez-vous [service] le [date]. Confirme l'annulation." |
| `modify_awaiting_datetime` | Modification requested | "Le client veut modifier son rendez-vous. Demande-lui la nouvelle date et heure." |
| `confirming_modify` | Confirm the new datetime | "Confirme le déplacement: [service] au [date] à [time]. Demande oui ou non." |
| `confirmed_modify_abandoned` | Modification declined | "Le client a refusé la modification. Son rendez-vous original est maintenu." |
| conflict | Slot taken | "Ce créneau est déjà pris. Demande-lui de choisir une autre heure." |
| fallback (greeting/unknown) | Default help | "Le client a écrit quelque chose d'incompréhensible. Redirige-le doucement." |

### Pattern 4: Graceful Degradation

The fallback map mirrors the current hardcoded strings in `machine.ts`. If `generateReply()` fails or times out, it returns from this map — ensuring silence never occurs.

```typescript
// Source: machine.ts analysis — mirrors existing hardcoded strings
const FALLBACK_REPLIES: Record<string, string> = {
  awaiting_service: "Quel service souhaitez-vous?",
  awaiting_datetime: "Quand souhaitez-vous votre rendez-vous?",
  confirming: "C'est bien ça? (oui/non)",
  confirmed: "Votre rendez-vous est confirmé.",
  cancelling: "Votre rendez-vous a été annulé.",
  modify_awaiting_datetime: "Quelle nouvelle date et heure souhaitez-vous?",
  confirming_modify: "Confirmer ce changement? (oui/non)",
  conflict: "Ce créneau est déjà pris. Voulez-vous choisir une autre heure?",
  default: "Je n'ai pas compris. Puis-je vous aider à réserver un rendez-vous?",
}
```

### Pattern 5: Timeout Implementation

`Promise.race` with `AbortSignal.timeout` is the correct Node.js pattern. The `@google/genai` SDK has known issues with `httpOptions.timeout` not being respected in all cases (confirmed via GitHub issues). `Promise.race` is more reliable.

```typescript
// Source: Node.js built-in AbortSignal, confirmed pattern
async function generateReply(context: ReplyContext): Promise<string> {
  const fallback = FALLBACK_REPLIES[context.nextStep] ?? FALLBACK_REPLIES.default

  try {
    const result = await Promise.race([
      callGeminiForReply(context),
      new Promise<never>((_, reject) =>
        setTimeout(() => reject(new Error('timeout')), 3000)
      ),
    ])
    return result ?? fallback
  } catch (err) {
    console.error('[reply] Gemini reply generation failed, using fallback:', err)
    return fallback
  }
}
```

### Pattern 6: Plain Text generateContent (distinct from intent extraction)

The intent layer uses `responseMimeType: 'application/json'` and `responseJsonSchema`. The reply layer uses neither — plain text output.

```typescript
// Source: @google/genai 1.48.0 type definitions + official docs
// response.text is a getter that concatenates all text parts from first candidate
async function callGeminiForReply(context: ReplyContext): Promise<string | undefined> {
  const ai = new GoogleGenAI({ apiKey: process.env.GEMINI_API_KEY! })
  const response = await ai.models.generateContent({
    model: 'gemini-2.5-flash',
    contents: [{ role: 'user', parts: [{ text: buildReplyPrompt(context) }] }],
    config: {
      systemInstruction: buildReplySystemPrompt(context.salonName),
      temperature: 0.7,       // some warmth, not robotic
      maxOutputTokens: 120,   // short WhatsApp messages, ~3 sentences max
    },
  })
  return response.text  // getter defined on GenerateContentResponse class
}
```

**Critical difference from existing `tryGemini()` in `models.ts`:**
- `models.ts` does `JSON.parse(result.text ?? '{}')` — structured output
- reply layer does `response.text` — plain string, no parsing

### Anti-Patterns to Avoid

- **Streaming for WhatsApp:** WhatsApp Cloud API `sendTextMessage` sends a complete string. There is no streaming delivery to end users. `generateContentStream` adds code complexity with no UX benefit.
- **Passing full conversation history to Gemini:** The project architecture explicitly avoids raw message history (STATE.md decision). The ReplyContext contains only the structured FSM fields.
- **Replacing `processIntent()` return value:** The FSM must still return `replyText` from hardcoded strings. Tests assert on `replyText` content (e.g., `expect(replyText).toContain('annule')`). Removing `replyText` from `ProcessResult` would break 30+ test assertions.
- **Calling Gemini for every FSM state unconditionally:** The `UPCOMING_BOOKINGS_PLACEHOLDER` path in `route.ts` handles query replies differently (formats from DB). That path should bypass `generateReply()` or be given its own context type.
- **Single Google GenAI client per request without timeout:** Gemini 2.5 Flash has been observed timing out at 120s on complex requests (AI Developers Forum issue). The 3s `Promise.race` is mandatory.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Token counting | Custom regex/word count estimator | `ai.models.countTokens()` (available in @google/genai) OR just set `maxOutputTokens: 120` | Token estimation is model-specific; `maxOutputTokens` caps the output and implicitly bounds cost and latency |
| Rate limiting | Custom token bucket | Rely on Gemini API 429 → catch → return fallback | For a single-tenant-per-request webhook, rate limits are unlikely at salon scale; build when needed |
| Response caching | LRU cache keyed on step+service+date+time | Not needed in Phase 4.5 | Same step + same booking details = same client, but different conversational tone is desirable. Caching would make replies feel robotic across conversations |
| Retry logic | Exponential backoff loop | Return fallback immediately | WhatsApp users expect a reply in <4s. A retry adds 1-3s. On Gemini failure, the fallback is the right answer, not a delayed retry. |

**Key insight:** The reply layer is a best-effort enhancement. The hardcoded FSM strings are not broken — they are the fallback. Any complexity that risks latency or reliability should be rejected.

## Common Pitfalls

### Pitfall 1: Modifying ProcessResult breaks existing tests
**What goes wrong:** If `processIntent()` is changed to not return `replyText`, all 30+ `fsm-machine.test.ts` assertions on `replyText` fail immediately.
**Why it happens:** The temptation to "clean up" the FSM return type when adding the reply layer.
**How to avoid:** Keep `ProcessResult` unchanged. The route calls `generateReply()` and uses its result, ignoring `processIntent()` return `replyText` (which now serves only as fallback source via the FALLBACK_REPLIES map).
**Warning signs:** Any edit to `machine.ts` return types during this phase is a red flag.

### Pitfall 2: `result.text` vs `response.text`
**What goes wrong:** Copying the pattern from `models.ts` (`result.text ?? '{}'`) and then trying to `JSON.parse()` a natural language reply.
**Why it happens:** `models.ts` uses `result.text` because the JSON mode returns a JSON string. The reply layer returns plain text.
**How to avoid:** The reply layer function returns `response.text` directly (the getter on `GenerateContentResponse`). No `JSON.parse()`. No `?? '{}'`. If `response.text` is `undefined`, fall back to the FALLBACK_REPLIES map.

### Pitfall 3: `httpOptions.timeout` not working in @google/genai
**What goes wrong:** Setting `new GoogleGenAI({ httpOptions: { timeout: 3000 } })` and assuming it enforces a 3s cap. It does not reliably do so (confirmed GitHub issue #1277).
**Why it happens:** The SDK has a known bug where `httpOptions.timeout` is not respected for `models.generateContent`.
**How to avoid:** Use `Promise.race` with `setTimeout` as the timeout mechanism, not SDK config.
**Warning signs:** Requests hanging past 3s in tests/production.

### Pitfall 4: Gemini generates mixed French/Darija unpredictably
**What goes wrong:** Darija words appear in every reply, making it feel forced. Or Gemini replies in English if the prompt is in English.
**Why it happens:** Without explicit language instruction, Gemini mirrors the input language.
**How to avoid:** System prompt must specify: "Réponds toujours en français. Tu peux ajouter une expression darija (inshallah, mabrook, wakha) au maximum une fois par conversation, et seulement si c'est naturel." The user prompt (context) should be in French, not English.

### Pitfall 5: Latency spike when bot_configs does not include salonName
**What goes wrong:** `generateReply()` receives `salonName: undefined`. The prompt has a missing field. Gemini invents a salon name.
**Why it happens:** `bot_configs` select in `route.ts` currently fetches only `active` and `owner_notification_wa_id`.
**How to avoid:** Add `salon_name` (or equivalent) to the `bot_configs` SELECT in `route.ts`, or use a safe default like `"le salon"` when the field is absent.

### Pitfall 6: UPCOMING_BOOKINGS_PLACEHOLDER path generates a reply twice
**What goes wrong:** `route.ts` query path currently builds `formattedReply` from DB data, bypassing `replyText`. If `generateReply()` is called before the PLACEHOLDER check, Gemini generates a reply that is immediately discarded.
**Why it happens:** The PLACEHOLDER check is currently a special case after `processIntent()`. If `generateReply()` is inserted above it, it runs unnecessarily.
**How to avoid:** Check for `replyText === 'UPCOMING_BOOKINGS_PLACEHOLDER'` BEFORE calling `generateReply()`. Only call `generateReply()` on normal FSM paths.

## Open Questions

1. **Does `bot_configs` contain `salon_name`?**
   - What we know: `route.ts` selects `active, owner_notification_wa_id` from `bot_configs`. The schema from Phase 1 includes a `system_prompt` column but the exact columns are not confirmed in this research.
   - What's unclear: Is there a dedicated `salon_name` column, or should it be derived from the `tenants` table?
   - Recommendation: Check `bot_configs` schema in Phase 1 migrations. If absent, add a JOIN on `tenants.name` in the `route.ts` query, or use `"ce salon"` as the default.

2. **Should `generateReply()` be called for the conflict case?**
   - What we know: Conflict replies are currently hardcoded in `route.ts` directly (not via `processIntent()`), so they do not flow through the normal `replyText` path.
   - What's unclear: Whether the warmth improvement is worth adding a separate `generateReply()` call in the conflict branch.
   - Recommendation: Yes, pass `conflict: true` in ReplyContext and handle it in `buildReplyPrompt()`. The conflict message is customer-facing and benefits from a warm, apologetic tone.

3. **Double LLM latency: is 4s target safe?**
   - What we know: Gemini 2.5 Flash TTFT ~0.5s, output ~0.8s for 80 tokens = ~1.3s per call. Two calls (intent + reply) = ~2.6s + network overhead.
   - What's unclear: Whether the two calls are sequential (adds latency) or can be parallelized (they cannot — reply needs `nextState` from intent + FSM first).
   - Recommendation: The 4s budget holds comfortably (2.6s + 1s overhead = 3.6s). Flag for monitoring. If intent escalates to OpenAI (~1.5-2s), total could reach 4-4.5s — still near the boundary. Log latency per message.

## Decisions Made

- Separate `tenants` query rather than JOIN — Supabase JS client is idiomatic with `.from()` chaining, not raw SQL
- Named reply variables per-path (`geminiReplyA`, `geminiReplyB`, etc.) — same function scope, no `const` shadowing
- `salonName` fetched only after `botConfig.active` check — avoids an extra DB call for inactive bots
- Owner notification message remains hardcoded (not Gemini) — it's a structured operational log, not a client-facing reply

**1. [Rule 1 - Bug] Fixed inline mock in duplicate-wamid test missing tenants chain**
- **Found during:** Task 2 verification (vitest run)
- **Issue:** The `skips processing duplicate wamid (count === 0)` test uses a custom inline `mockFrom.mockImplementation`. Its fallback returned `{ select: () => ({ single: () => ... }) }` — no `.eq()` method. The new `tenants` query in route.ts calls `.select().eq().single()`, causing `TypeError: supabase.from(...).select(...).eq is not a function`.
- **Fix:** Added explicit `tenants` case to the inline mock, and fixed the fallback to include `.eq()` in the chain: `{ select: () => ({ eq: () => ({ single: () => ... }) }) }`
